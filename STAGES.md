# Этапы реализации PWA «Карточки для изучения слов»

Каждый этап — отдельный шаг. После этапа можно остановиться, всё должно работать.

---

## Принципы вёрстки и архитектуры

**Чтобы потом легко менять дизайн:**

- Вся разметка — **семантичный HTML** (`<article>`, `<section>`, `<nav>`, `<button>` и т.п.)
- Стили — **Tailwind CSS** (классы), без inline-стилей
- **Много мелких компонентов**: один компонент = одна зона экрана или один UI-элемент
- Каждый компонент в отдельном файле в `components/`
- Логика и данные — в `lib/`, `hooks/`, страницы только собирают компоненты
- Имена компонентов по смыслу: `TranslateInput`, `CardListItem`, `DeckProgressBar` и т.д.

## Этап 1: Каркас проекта ✅
**Цель:** Next.js + Tailwind, базовая структура.

- Next.js (App Router) + TypeScript + Tailwind
- Layout с нижней навигацией
- Заглушки страниц

**Результат:** приложение запускается, можно переключать вкладки.

---

## Этап 2: Типы, пользователь и localStorage
**Цель:** Модель данных с учётом пользователя, готовность к будущей регистрации.

- **Типы:** `Card`, `Deck`, `TranslationHistoryItem`, `User`
- **Пользователь:** при первом заходе генерируем `userId` (UUID), сохраняем в `localStorage`
- Все данные привязаны к `userId`: ключи вида `translator-{userId}-decks`, `translator-{userId}-history` и т.п.
- Функции в `lib/storage.ts`: `getCurrentUser()`, `getDecks()`, `getHistory()`, `saveCard()`, создание дефолтной колоды «All cards»
- Структура `Card`: `id`, `foreign`, `translation`, `customTranslation?`, `imageUrl?`, `examples?`, `deckIds`, `learned`, `createdAt`

**Результат:** данные «привязаны» к пользователю, в будущем можно заменить localStorage на API с тем же `userId`.

---

## Этап 3: Вкладка «Перевод»
**Цель:** Ввод текста → перевод через MyMemory → показ результата.

- Компоненты: `TranslateInput`, `TranslateResult`, `TranslatePage` (собирает их)
- Запрос к MyMemory API, отображение перевода
- Сохранение в историю текущего пользователя

**Результат:** ввёл слово — видишь перевод, запись в истории.

---

## Этап 4: Сохранить как карточку + свой перевод
**Цель:** Сохранение карточки с возможностью ввести свой перевод.

- Компоненты: `SaveCardButton`, `DeckSelect`, `CustomTranslationInput`
- При сохранении: выбор колоды + опциональное поле «Свой перевод» (перезаписывает или дополняет API-перевод)
- В `Card`: `translation` — из API, `customTranslation` — от пользователя; при показе используем `customTranslation || translation`
- То же для элементов истории: кнопка «Сохранить как карточку» + ввод своего перевода

**Результат:** можно сохранять карточки с переводом из API или своим.

---

## Этап 5: Картинка к слову
**Цель:** У каждой карточки — картинка по смыслу из открытого источника.

- API: Unsplash (бесплатный ключ) или Pexels (бесплатный ключ) — поиск по слову/фразе
- При сохранении карточки или при первом показе — запрос картинки, сохранение `imageUrl` в `Card`
- Компонент `CardImage` — отображение картинки с fallback (плейсхолдер, если нет картинки или ошибка)
- Кэш в localStorage или в объекте карточки

**Результат:** у карточки есть картинка, соответствующая смыслу слова.

## Этап 11: Сохранение и восстановление данных

- **Экспорт:** кнопка «Скачать резервную копию» — выгрузка всех данных (карточки, колоды, история, userId) в JSON-файл

**Результат:** пользователь может сохранить данные на диск и восстановить их при необходимости.

## Этап 14: Мягкая авторизация (защита API от ботов)

**Цель:** Защитить `/api/translate` от злоупотреблений, не ломая UX для гостей.

**Подход — два шага:**

### Шаг 14.1 — Лимит для гостей
- Гость может делать **N переводов бесплатно** (например, 20 в день)
- Счётчик хранится в `localStorage` с датой сброса
- После лимита — ненавязчивый баннер: «Войдите, чтобы переводить без ограничений»
- Функция `lib/usageLimit.ts`: `canTranslate()`, `incrementUsage()`, `resetIfNewDay()`

### Шаг 14.2 — Rate limiting на уровне API
- Установить `@upstash/ratelimit` + `@upstash/redis` (бесплатный тариф)
- В `/api/translate`: проверять IP или `userId`
  - Гость по IP: до **30 запросов/час**
  - Авторизованный по userId: до **200 запросов/час**
- При превышении — HTTP 429 с сообщением об ошибке

**Компоненты для разработки:**
- `lib/usageLimit.ts` — клиентский счётчик
- `components/TranslationLimitBanner.tsx` — баннер с призывом зарегистрироваться
- Обновить `app/api/translate/route.ts` — добавить rate limit check

**Что НЕ меняется:**
- Middleware остаётся публичным (гости заходят свободно)
- Структура данных не меняется
- Clerk остаётся опциональным

**Результат:** боты получают 429, реальные пользователи видят мягкое предложение зарегистрироваться, зарегистрированные — без ограничений.

- `userId` генерируется при первом заходе, хранится в `localStorage`
- Все данные ключаются по `userId`

Позже (регистрация):
- Заменить `getCurrentUser()` на вызов API: при логине возвращается `userId`
- Миграция: при первом логине предложить «перенести локальные данные» — отправить `localStorage` на сервер, привязать к аккаунту
- Структура данных (`Card`, `Deck` и т.д.) остаётся той же, меняется только слой хранения (API вместо localStorage)
